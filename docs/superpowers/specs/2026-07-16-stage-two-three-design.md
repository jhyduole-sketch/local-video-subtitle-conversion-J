# 阶段二与阶段三优化设计

## 目标

在不改变现有 CLI 与任务历史兼容性的前提下，降低模型配置和 Web 代码的重复度，并提高本地 NLLB 长视频翻译的速度、可恢复性和进度可见性。

## 阶段二：结构与界面精简

1. 建立统一翻译引擎目录，集中保存引擎 ID、兼容别名、显示名称和 NLLB 模型名称。Pipeline、CLI 和健康检查从同一处读取，避免名称和路由漂移。
2. 把 Web 环境检查从 `web.py` 提取为独立模块，同时在 `web.py` 保留兼容导出，现有调用和测试不需要迁移。
3. 把前端语言目录从 `app.js` 提取到独立静态资源，`app.js` 只保留交互逻辑。
4. 将模型路径、VAD路径、字幕编码和字幕位置放入“高级设置”，默认界面保留输入、语言、转写、翻译和输出开关。

## 阶段三：性能与长任务恢复

1. NLLB 根据模型、设备和内存选择安全批次；用户设置 `LOCAL_TRANSLATION_BATCH_SIZE` 时仍以用户值为准。
2. 批次遇到内存不足时自动减半重试，最低降到单条；其它错误仍立即抛出，不掩盖模型质量问题。
3. NLLB 每完成一个批次就保存部分翻译，任务停止或异常后可以从已完成字幕继续。
4. `local-nllb` 与 `local-nllb-quality` 使用统一缓存身份，旧命令和新页面共享1.3B翻译缓存。
5. NLLB进度日志显示完成条数、已用时间和预计剩余时间。

## 稳定性边界

- 不并行加载多个1.3B模型，不同时翻译多个目标语言，避免内存争抢。
- 不自动删除旧600M缓存。
- 不改变输出文件命名、任务目录和字幕格式。
- 不提交阶段二、三代码，完成后等待用户测试确认。

## 验收

- Web基础流程与历史任务继续可用。
- NLLB中断后重新执行只处理缺失字幕。
- 模拟内存不足时批次自动缩小。
- 全量单元测试、Python编译、JavaScript语法和服务健康检查通过。
